1. hco_pipeline.py
   ─────────────────────────────────────────────────────────────────────────

   FIX C — Locality Strip (Critical bug fix)
   ─────────────────────────────────────────
   PROBLEM:
     Veeva India HCO names follow "HCO Name - Locality" convention.
     norm_text() converts " - " to a space before _strip_locality() runs.
     Result: city was baked into name_norm as the last token.
     "Dr M Rashid's Clinic - Malda" vs "Dr B K Ghosh's Clinic - Malda"
     scored 76.6% name similarity — far above the 75% review gate.
     This caused 33,149 false-positive REVIEW pairs on the Sanofi BC dataset.

   FIX:
     Split raw name on " - " BEFORE calling norm_text():
       raw_name_for_cmp = raw_name.split(" - ")[0].strip()
       name_stripped_norm = norm_text(raw_name_for_cmp)
     Store name_stripped_norm in canonical record.
     _compute_hco_signals() reads name_stripped_norm instead of running
     _strip_locality() on the already-normalized string.

   IMPACT: Run2 REVIEW pairs: 33,149 → 2,037 (−94% false positives).

   CHANGE TYPE: Bug fix in _build_hco_canonical() and _compute_hco_signals()
   BREAKING CHANGES: None — name_stripped_norm falls back to _strip_locality()
                     if field is absent (backward compatible)

   ─────────────────────────────────────────────────────────────────────────
   ACRONYM DETECTION (New feature)
   ─────────────────────────────────────────────────────────────────────────
   PROBLEM:
     "GSV Medical College" vs "Ganesh Siddha Venkateswara Medical College"
     score only ~40% name similarity. Same address, same type — genuine dupe
     missed by all existing rules.

   FIX:
     Added is_acronym_of(short_name, long_name) function.
     Detects when the first token of the short name is the initials of
     meaningful words in the long name.
     Examples: GSV→Ganesh Siddha Venkateswara, AIIMS→All India Institute
     of Medical Sciences, KEM→King Edward Memorial.

   NEW AUTO RULES:
     H4: Acronym match + Address overlap + Type match → AUTO-MERGE
     H5: Acronym match + Address overlap + Phone/Fax match → AUTO-MERGE

   REVIEW SCORING:
     acronym_match: +20 points added to _apply_hco_review_scoring()

   CHANGE TYPE: Addition (new function + 2 new auto rules + scoring signal)
   BREAKING CHANGES: None

   ─────────────────────────────────────────────────────────────────────────
   INDIAN ADDRESS STOP WORDS (New feature)
   ─────────────────────────────────────────────────────────────────────────
   PROBLEM:
     Original stop word list was entirely Western (street, avenue, boulevard).
     Indian address terms (nagar, salai, marg, chowk, colony, sector, road,
     lane, cross, main, layout, extension, phase, block, plot) were counted
     as meaningful discriminating tokens, causing false address overlaps.

   FIX:
     Replaced Western stop words with comprehensive Indian address stop words.

   ADDRESS TOKEN OVERLAP THRESHOLD:
     Tightened from 50% → 70% to reduce noise matches.

   CHANGE TYPE: Replacement of stop word list + threshold adjustment
   BREAKING CHANGES: None

   ─────────────────────────────────────────────────────────────────────────
   ADDRESS AS MANDATORY GATE (New behavior)
   ─────────────────────────────────────────────────────────────────────────
   PROBLEM:
     Without an address gate, pairs with similar names at different locations
     entered REVIEW — consuming manual review time for clear non-duplicates.
     Two HCOs at different addresses cannot be duplicates regardless of name.

   FIX:
     address_match is now a mandatory prerequisite for REVIEW entry.
     Pairs that don't meet the 70% address token overlap threshold are
     classified NOT-DUP immediately, skipping the scoring phase.

   CHANGE TYPE: Behavioral — adds mandatory gate in _apply_hco_review_scoring()
   BREAKING CHANGES: None (conservative change — only removes false positives)

   ─────────────────────────────────────────────────────────────────────────
   REVIEW THRESHOLD RAISED: 50 → 70 (Tuning)
   ─────────────────────────────────────────────────────────────────────────
   PROBLEM:
     At threshold 50, pairs with only city + state + type match (45 pts) plus
     minimal name signal could enter REVIEW. These were noise, not genuine
     duplicate candidates.

   FIX:
     review_threshold: 50 → 70

   IMPACT: REVIEW pairs reduced from 48,227 → 18,662 on the Sanofi BC dataset
           (further reduced to 2,037 after Fix C).

   ─────────────────────────────────────────────────────────────────────────
   SCORING REWEIGHT — Address-Priority Model (Tuning)
   ─────────────────────────────────────────────────────────────────────────
   Rationale: For HCOs, address is ground truth. Name confirms; address decides.

   OLD vs NEW weights:
   ┌─────────────────────────┬─────┬─────┐
   │ Signal                  │ Old │ New │
   ├─────────────────────────┼─────┼─────┤
   │ address_match           │  30 │  50 │
   │ postal_match            │  25 │  30 │
   │ city_match              │  20 │  10 │
   │ name_strong (≥92%)      │  35 │  25 │
   │ name_medium (≥85%)      │  25 │  15 │
   │ phone_match             │  20 │  25 │
   │ different_city penalty  │ −15 │ −30 │
   └─────────────────────────┴─────┴─────┘

   CHANGE TYPE: Configuration tuning in _apply_hco_review_scoring()
   BREAKING CHANGES: None

2. hcp_pipeline.py
   
   CHANGE 1: _compute_signals() signature
   OLD: def _compute_signals(ar: dict, br: dict, shared_detector: SharedContactDetector) -> dict[str, Any]:
   NEW: def _compute_signals(ar: dict, br: dict, shared_detector: SharedContactDetector | None = None) -> dict[str, Any]:
   
   REASON: Worker processes can't serialize SharedContactDetector objects
   IMPACT: shared_detector now optional, defaults to None (safe for multiprocessing)
   
   CHANGE 2: _compute_signals() shared contact handling
   OLD: shared = shared_detector.is_shared(phones_ov, emails_ov)
   NEW: shared = shared_detector.is_shared(phones_ov, emails_ov) if shared_detector else False
   
   REASON: Prevent crashes when shared_detector is None
   IMPACT: In large-scale mode, shared contact detection is disabled (acceptable)
   
   CHANGE 3: _enrich_row() signature
   OLD: def _enrich_row(vid_a: str, vid_b: str, canon_idx: pd.DataFrame) -> dict:
   NEW: def _enrich_row(vid_a: str, vid_b: str, canon_idx: pd.DataFrame | dict) -> dict:
   
   REASON: run_large.py uses dict-based index (more compact than DataFrame)
   IMPACT: Function now accepts both DataFrame (run.py) and dict (run_large.py)
   
   CHANGE 4: _enrich_row() implementation
   - Added isinstance() checks to handle both DataFrame and dict inputs
   - Uses .get() method for dict access (safe, returns empty string if missing)
   - Maintains backward compatibility with DataFrame-based indices
   
   REASON: Enable worker processes to use plain dict canonical indices
   IMPACT: run.py continues to work unchanged, run_large.py can use dicts
   
   BREAKING CHANGES: None (fully backward compatible)
   
   AFFECTED FUNCTIONS:
   • run.py — No changes needed, works as before
   • run_large.py — Can now use multiprocessing with dict indices

MIGRATION PATH
═════════════════════════════════════════════════════════════════════════════

Small datasets (<300K rows):
  → Continue using: python -m hcp_dupe_tool.run --input data.xlsx --outdir ./out

Large datasets (600K+ rows):
  → Use new tool: python -m hcp_dupe_tool.run_large --input data.csv --outdir ./out

Switching between modes:
  1. No code changes required
  2. Input format differs:
     - run.py: XLSX or CSV (faster with XLSX)
     - run_large.py: CSV only (required)
  3. Output directory can be same or different
  4. Config can be reused (run_large.py has sensible defaults)

Targeted for:
  • 600K+ raw rows
  • 300K+ unique VIDs
  • 8-16 GB RAM
  • CSV input format
